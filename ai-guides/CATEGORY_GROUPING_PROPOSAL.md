# Category Grouping Proposal - Top-Level Organization

## Research: What Other Apps Do

### Common Approaches:

1. **YNAB (You Need A Budget)**
   - Groups by: Immediate Obligations, True Expenses, Quality of Life Goals, Just for Fun
   - Philosophy: Based on spending priorities and financial goals

2. **Mint**
   - Groups by: Bills & Utilities, Shopping & Entertainment, Food & Dining, Auto & Transport, Travel, Health & Fitness, etc.
   - Philosophy: By life area/activity

3. **PocketGuard**
   - Groups by: Needs, Wants, Savings & Debt
   - Philosophy: Simple needs vs wants

4. **Goodbudget**
   - Groups by: Monthly Bills, Everyday Expenses, Savings Goals, Debt Payments
   - Philosophy: By expense type and priority

5. **Simplifi**
   - Groups by: Bills & Utilities, Shopping & Entertainment, Food & Dining, etc.
   - Philosophy: Similar to Mint - by life area

## Implementation Considerations

### Database Schema
We could add a `group` field to categories:
```prisma
model categories {
  // ... existing fields
  group String? // "income", "essential", "lifestyle", "financial", "other"
}
```

Or create a separate `category_groups` table:
```prisma
model category_groups {
  id          Int          @id @default(autoincrement())
  name        String       @unique
  description String?
  sort_order  Int          @default(0)
  categories  categories[]
}

model categories {
  // ... existing fields
  group_id    Int?
  group       category_groups? @relation(fields: [group_id], references: [id])
}
```

### UI Display
- Show groups as collapsible sections
- Allow filtering by group
- Show spending totals per group
- Visual indicators (colors, icons) for each group

### User Customization
- Allow users to move categories between groups
- Or create custom groups
- Default groups are suggestions, not fixed

## Alternative: No Top-Level Groups

**Option 6: Keep Current Structure**
- Just show all categories in a flat list or organized by parent/child
- Let users create their own organization via custom categories
- Simpler implementation
- More flexible for users

**Pros:**
- Simpler to implement
- No assumptions about user priorities
- Users can organize as they see fit

**Cons:**
- Less guidance for new users
- Harder to see spending patterns at a glance
- No built-in budgeting framework

## My Recommendation

**Go with Option 1 (Needs vs Wants) as a soft grouping:**
- Add a `group` field to categories (simple string, no new table needed)
- Make it optional - categories can exist without a group
- Use groups for:
  - Visual organization in category list
  - Spending reports ("Essential vs Lifestyle")
  - Budgeting guidance
- Allow users to customize group assignments if needed
- Keep it simple - don't over-engineer

**Groups:**
1. `income` - All income categories
2. `essential` - Needs (housing, food, transport, healthcare, etc.)
3. `lifestyle` - Wants (dining out, shopping, entertainment, etc.)
4. `financial` - Savings, investments, transfers
5. `other` - Uncategorized, misc

This gives structure without being too rigid.
